# html的渲染过程

我们知道html页面含有html标签, css样式, js脚本, 那么浏览器具体怎么组织它们, 才最终绘制我们所看到的 页面呢?

首先了解一个事件: DOMContentLoaded, 该事件会在html解析完成之后触发

3.当有JS时，HTML文档解析过程为：

可以看到, html解析过程中先遇到css, 此时并行解析css, html继续解析, 然后遇到js脚本, 下载js脚本会阻塞html的解析 ,js下载完成后必须等待前面所有css资源解析完毕才会执行, css解析完成后, js脚本开始执行, js脚本执行完成后, html继续解析, html解析完成后, 与CSSOM 一起构成layoutTree, 浏览器由此才开始绘制页面

综上:

  • css解析不会阻塞html解析
  • css解析会阻塞html渲染
  • js加载执行会阻塞html的解析和渲染
  • css解析会阻塞后面js的执行

由此看到, css 解析过程中, 如果紧接有js代码, 则js代码执行被阻塞, 进而html解析完全停住, Dom最后才形成, 因此, 页面此时才开始绘制.因此, 一般建议将js代码放置在文档最后, 尽量加快dom和css的解析

# 既然js脚本下载都会阻塞html的解析, 那能不能并行下载js脚本呢?

当然可以, script标签的defer和async属性可以让js异步下载, 下载过程中html可以正常解析

# defer

defer属性可以异步加载js资源, 并且js资源下载完成后并不会立即执行, 而是会等待html解析完毕, DOMContentLoaded事件执行之前执行

1.HTML还没解析完成时，defer脚本已经加载完毕，那么defer脚本将等待HTML解析完成后再执行。 defer脚本执行完毕后触发DOMContentLoaded事件。如下图所示

2.HTML解析完成时，defer脚本还没加载完毕，那么defer脚本继续加载，加载完成后直接执行，执行完毕后触发DOMContentLoaded事件。如下图所示：

总之, defer脚本会在DOMContentLoaded事件之前执行

# async

async属性也会让js资源异步加载, 且一旦js资源加载完毕, 会立即执行, 且js脚本执行顺序不一定是script标签书写顺序

1.HTML 还没有被解析完的时候，async脚本已经加载完了，那么 HTML 停止解析，去执行脚本，脚本执行完毕后触发DOMContentLoaded事件。如下图所示：

2.HTML 解析完了之后，async脚本才加载完，然后再执行脚本，那么在HTML解析完毕、async脚本还没加载完的时候就触发DOMContentLoaded事件。如下图所示：

注意

defer 和 async 均存在兼容性问题, async仅支持 > IE9, defer虽支持IE9以下, 但仍然存在某些bug, 因此, 最好的方法就是 将js脚本放置在body后面
